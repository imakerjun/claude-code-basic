# CLAUDE.md 예시 모음

조직에 들이고 싶을 때 참고할 수 있는 한 줄 규칙들입니다. **그대로 복사하지 말고** 본인 조직의 맥락에 맞게 다듬어 사용하십시오.

## 품질·톤 관련

- 한국어 문서는 `~합니다` 체를 쓰고 `~요` 체는 쓰지 않는다.
- 숫자에는 반드시 출처를 명시한다.
- 한 줄 요약이 없는 제안서·보고서는 생성하지 않는다.
- 기획 문서의 첫 줄은 "왜" 로 시작한다.
- 회의 아젠다는 최대 5개 항목까지만 작성한다.
- 모든 의사결정 요청에는 "안 하는 선택" 이 비교안으로 포함된다.
- 제품명·서비스명은 공식 표기만 사용한다 (예: 배달의민족, 배민페이).

## 보안·컴플라이언스 관련

- 고객 데이터가 포함된 분석은 반드시 익명화부터 한다.
- 민감 필드(이메일·전화번호·주민번호·카드번호)는 출력에 포함하지 않는다.
- 결제·정산 관련 SQL은 읽기 전용 계정 기준으로 작성한다.
- 외부 API 호출이 발생하는 코드는 로그 기록 예시를 함께 제시한다.
- PG사 API 키·시크릿은 예시 코드에도 절대 포함하지 않는다.

## 프로세스 관련

- 모든 SQL은 실행 계획과 예상 행 수를 함께 제시한다.
- 커밋 메시지는 [JIRA-번호]로 시작한다.
- PR 설명은 "왜 이 변경이 필요한가" 가 맨 위에 온다.
- 장애 보고서는 타임라인 → 원인 → 대응 → 재발 방지 순서를 따른다.
- 회의록은 "결정사항"과 "액션 아이템"을 반드시 분리한다.

## 도메인 특화 — 결제 플랫폼 예시

- 결제 금액 계산은 모두 정수형(원 단위)으로 다루고 실수형을 쓰지 않는다.
- PG사 이름을 드러내야 하는 경우 "토스페이먼츠·KCP·나이스페이" 공식 명칭을 쓴다.
- 정산·수수료 관련 계산은 반드시 검증 테스트 케이스를 동반한다.
- 결제 실패율·승인 지연 같은 지표는 p50이 아닌 p95·p99로 이야기한다.

## 도메인 특화 — 운영·CS 예시

- CS 티켓 카테고리는 기존 5개(결제/배송/쿠폰/계정/기타) 외를 만들지 않는다.
- 사장님 대상 안내 문구는 반드시 존댓말 최종 검수를 거친다.
- 운영 공지는 영업시간 기준으로 시각을 표기한다.

## 한 줄 규칙을 만드는 팁

1. **금지 < 강제**: "하지 말자" 보다 "하자" 가 강력합니다.
2. **모호한 형용사 피하기**: "깔끔하게" 대신 "3줄 이내로".
3. **위반 시 어떻게 되는지**: 자동 차단(Hooks)인지, 가이드(CLAUDE.md)인지 명확히.
4. **한 번에 한 줄**: 처음부터 10줄 만들지 말고, 한 줄로 시작해서 늘립니다.
5. **팀에 물어보기**: 혼자 결정하지 말고, 팀이 매일 부딪히는 문제를 먼저 들어봅니다.
