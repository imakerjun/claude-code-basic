# 4. 도입 시 고려사항 (10분)

T3 개념을 이해했다면 이제 **조직에 들일 때 짚어둘** 네 가지를 확인합니다.

## 1. 보안

### 원칙

**민감 데이터는 경계선을 긋는다.**

### 질문 체크리스트

- 어떤 저장소에 Claude Code를 허용할 것인가?
- 고객 데이터가 들어 있는 저장소는 어떻게 취급하는가?
- API 키·비밀번호가 담긴 파일을 읽지 못하게 막을 수 있는가? (Hooks 활용)
- 외부 API 호출이 발생하는 명령을 어떻게 통제하는가?

### 우선 조치

- 민감 저장소는 Claude Code 허용 범위에서 제외
- `.gitignore`와 동일하게 Claude Code가 읽지 말아야 할 경로 정의
- 감사 로그 활성화

## 2. 품질

### 원칙

**검증 없이 AI 결과를 수용하지 않는다.**

### 질문 체크리스트

- AI가 만든 결과물을 누가 리뷰하는가?
- "Claude가 그렇다고 했어요" 가 정당화되는가? — **안 된다**
- 최종 책임은 누구인가? — 사람

### 우선 조치

- "AI 생성물에는 반드시 사람의 리뷰 흔적이 남는다" 를 CLAUDE.md에 명시
- 중요도에 따른 리뷰 레벨 정의 (간단한 요약 vs 고객 문서)

## 3. 팀 온보딩

### 원칙

**CLAUDE.md는 팀 문서의 일부로 관리한다.**

### 질문 체크리스트

- 신규 입사자가 CLAUDE.md를 언제 읽는가?
- CLAUDE.md를 업데이트할 권한은 누구에게 있는가?
- 변경 이력을 어떻게 추적하는가?

### 우선 조치

- CLAUDE.md를 **팀 공식 문서 시스템** 의 일부로 둔다
- 변경은 PR 리뷰를 거친다
- 월 1회 리뷰 미팅으로 "지난 한 달 동안 CLAUDE.md에 추가/수정된 것" 공유

## 4. 거버넌스

### 원칙

**라이선스·저작권·감사 추적을 명시한다.**

### 질문 체크리스트

- Claude Code 사용 라이선스 체계는? (팀 플랜, 엔터프라이즈 플랜)
- AI 생성물의 저작권 귀속은 어떻게 되는가?
- 규제 산업(금융·의료)에 해당하는 경우 감사 추적이 충분한가?

### 우선 조치

- 법무/컴플라이언스 팀과 사전 협의
- "감사 추적이 필요한 작업" 리스트 작성
- 필요 시 Hooks로 자동 로깅

## 요약

| 축 | 한 줄 원칙 | 주요 재료 |
|---|---|---|
| 보안 | 민감 데이터는 경계를 긋는다 | Hooks |
| 품질 | 검증 없이 수용하지 않는다 | CLAUDE.md, Agents |
| 팀 온보딩 | CLAUDE.md는 팀 문서의 일부 | CLAUDE.md |
| 거버넌스 | 라이선스·감사 추적 명시 | Hooks |

첫 번째 실천은 **네 가지 중 하나를 골라 팀에 질문을 던지는 것** 입니다.

> **"우리 팀은 이 네 가지 중 어디가 가장 취약한가?"**
