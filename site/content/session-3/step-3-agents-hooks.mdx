# 3. Agents와 Hooks · 개념 소개 (5분)

> 이 두 가지는 **개념만** 소개합니다. 90분 입문 세션에서 깊이 들어가지 않습니다.

## Agents · 역할 기반 위임

### 한 문장 정의

> **Agents는 특정 역할을 맡은 전문 AI 일꾼들입니다. 각자 다른 CLAUDE.md와 다른 Skills를 가질 수 있습니다.**

### 예시

- **코드 리뷰어 에이전트** — 풀 리퀘스트를 받으면 품질·보안 관점으로 검토
- **회의록 요약 에이전트** — 긴 회의록을 3줄로 줄임
- **고객 이슈 분류 에이전트** — 들어온 문의를 카테고리로 나눔

### 왜 중요한가

팀에 서로 다른 전문가가 있는 것과 같은 구조를 AI에도 만들 수 있다는 뜻입니다. **역할별로 다른 규칙을 따르게 할 수 있습니다.**

예를 들어:
- "코드 리뷰어" 는 **보수적** 이어야 합니다. 의심스러우면 지적하는 게 낫습니다.
- "아이디어 브레인스토머" 는 **관대** 해야 합니다. 의심스러워도 일단 꺼내는 게 낫습니다.

둘은 반대되는 성향입니다. 한 AI에게 둘 다 시키면 애매해집니다. Agents로 나누면 각각 명확해집니다.

## Hooks · 정책의 자동 집행

### 한 문장 정의

> **Hooks는 특정 행동 전/후에 자동으로 실행되는 검사 장치입니다. '지켜주세요' 가 아니라 '못 하게 막는다' 입니다.**

### CLAUDE.md와의 차이

| | CLAUDE.md | Hooks |
|---|---|---|
| 성격 | 가이드라인 ("지켜주세요") | 강제 집행 ("못 하게 막는다") |
| 위반 시 | 모델이 잊을 수도 있음 | **기계적으로 차단** |
| 용도 | 품질·톤 | 보안·컴플라이언스 |

### 예시

- **커밋 전 민감 정보 스캔** — API 키가 들어 있으면 자동 차단
- **배포 전 테스트 통과 확인** — 테스트 실패 시 배포 불가
- **고객 데이터 접근 시 로그 기록** — 누가 언제 어떤 데이터를 봤는지 감사 추적

### 왜 중요한가

조직에 **강제력이 필요한 규칙** 이 있습니다. 예를 들어 "고객 주민번호는 절대 로그에 남기지 않는다" 같은 규칙은 **'가이드'로는 부족합니다.** 어기면 사고가 납니다.

이런 규칙은 Hooks로 자동 집행해야 합니다.

## 네 가지 재료의 관계

```
CLAUDE.md  →  원칙과 가이드
Skills     →  반복 절차의 이름
Agents     →  역할별 전문 일꾼
Hooks      →  강제력 있는 자동 검사
```

임원진 관점에서 보면 네 가지 모두 **"조직이 AI와 일하는 방식을 설계하는 재료"** 입니다.

오늘 여러분이 가지고 가셔야 할 것은 이것입니다.

> **"나는 이 네 가지 재료로 우리 조직의 AI 사용 방식을 설계할 수 있다."**
