# T3 하네스 엔지니어링 라이브 데모 대본

20분 섹션. **이 교육의 정점**이다. 여기서 "이것이 내 레버다"가 나오지 않으면 세션 실패.

## 핵심 메시지 (매 데모마다 반복)

> **"여러분은 코드를 짤 필요가 없습니다. 여러분은 규칙을 설계합니다."**

---

## 1. CLAUDE.md 라이브 데모 (10분)

### 목적

조직의 AI 동작 기준을 **한 파일**로 바꿀 수 있다는 것을 실시간으로 목격하게 한다.

### 준비

세션 시작 전 `CLAUDE.md`를 다음과 같이 최소 상태로 되돌려둔다.

```markdown
# CLAUDE.md

이 저장소는 우아한형제들 임원진 대상 Claude Code 90분 입문 교육 자료입니다.
```

### 장면 1 — 빈 상태에서 요청 (3분)

**프롬프트**:
```
다음 주 임원회의 아젠다 초안을 3줄로 만들어줘. sample-team 근거 문서는 안 봐도 돼.
```

Claude는 일반적인 톤으로 아젠다를 생성한다. 여러 형식 가능성이 열려 있다.

**강사 멘트**: "일반적인 결과입니다. 우리 조직의 규칙은 아무것도 반영되지 않았습니다."

### 장면 2 — CLAUDE.md에 두 줄 추가 (3분)

에디터에서 `CLAUDE.md`에 다음 두 줄을 추가하고 저장한다. 수강자가 보는 화면에서 직접 타이핑한다.

```markdown
## 조직 규칙

- 모든 응답은 존댓말로 작성한다.
- 회의 아젠다는 최대 5개 항목까지만 작성한다.
- 각 아젠다 항목은 한 줄을 넘지 않는다.
```

**강사 멘트**: "보세요. 제가 지금 타이핑한 이 세 줄이 오늘부터 이 조직의 AI 동작 기준입니다."

### 장면 3 — 동일 요청 반복 (3분)

**동일한 프롬프트**를 다시 친다.

```
다음 주 임원회의 아젠다 초안을 3줄로 만들어줘. sample-team 근거 문서는 안 봐도 돼.
```

Claude가 존댓말, 한 줄 제약을 지킨 결과를 출력한다. **이 차이가 수강자 눈앞에서 일어나는 것이 핵심**이다.

**강사 멘트**: "같은 사람, 같은 요청입니다. 차이는 단 하나 — CLAUDE.md에 세 줄을 적은 것뿐입니다. **이 파일이 여러분이 말한 규칙을 조직 전체의 AI가 따르게 만드는 장치입니다.** 이것이 하네스 엔지니어링의 첫 번째 요소입니다."

### 장면 4 — 응용 아이디어 1분

**강사 멘트**: "예를 들어 이런 것들을 CLAUDE.md에 넣을 수 있습니다."

- "모든 SQL 쿼리는 쓰기 전에 팀장 리뷰를 받는다"
- "고객 데이터가 포함된 분석은 반드시 익명화부터 한다"
- "한국어 문서는 `~합니다` 체를 쓰고 `~요` 체를 쓰지 않는다"
- "커밋 메시지는 [JIRA-번호]로 시작한다"

**강사 멘트**: "이것이 '시스템을 설계한다'의 의미입니다. 여러분 각자가 조직에 들이고 싶은 한 줄이 있을 겁니다. 그것이 여러분의 CLAUDE.md 초안입니다."

## 2. Skills 라이브 데모 (5분)

### 목적

반복되는 워크플로우를 **이름으로 부르는** 개념을 이해시킨다.

### 준비

`expert-panel` skill 호출 장면 녹화본 (또는 교육 설계에 실제로 사용한 기록)을 준비한다.

### 장면 1 — 개념 설명 (2분)

**강사 멘트**: "CLAUDE.md가 '항상 적용되는 규칙' 이라면, Skills는 '필요할 때 부르는 이름 붙은 절차' 입니다. 예를 들어 '코드 리뷰', '고객 이슈 분류', '회의록 정리' 같은 반복 작업을 이름으로 만들어두는 것입니다."

### 장면 2 — 실제 예시 (3분)

강사는 **이 교육 자료를 만들 때 실제로 사용한** skill을 보여준다.

**강사 멘트**: "이 교육 커리큘럼을 설계할 때 저는 `전문가소환` 이라는 skill을 썼습니다. '전문가 6명을 불러서 근거 기반으로 토론해줘' 라는 워크플로우에 이름을 붙여둔 거죠. 한 번 만들어두면 다음부터는 이름만 부르면 됩니다."

- 화면에 skill 파일 구조 보여주기 (읽기 전용)
- 호출 기록 보여주기

**강사 멘트**: "조직 지식의 코드화입니다. 여러분 조직에 이런 '반복되는 의사결정 절차' 가 뭐가 있는지 한번 떠올려보세요. 그게 여러분의 Skills 후보입니다."

## 3. Agents 언급 (3분)

### 목적

개념만 소개. 라이브 데모는 생략.

**강사 멘트**:

"세 번째 요소는 Agents입니다. **역할 기반 위임**입니다."

- "코드 리뷰어 에이전트" — 풀 리퀘스트를 검토하는 전문 역할
- "보안 검토자 에이전트" — 민감 정보 포함 여부 확인
- "회의록 요약 에이전트" — 긴 회의록을 3줄로

"중요한 점은 이것입니다. 각 에이전트는 **서로 다른 CLAUDE.md와 다른 Skills를 가질 수 있습니다.** 즉, 역할별로 다른 규칙을 따릅니다. 팀에 전문가가 있듯, AI에도 전문가를 둘 수 있다는 뜻입니다."

---

## 4. Hooks 언급 (2분)

### 목적

개념만 소개. 라이브 데모는 생략.

**강사 멘트**:

"마지막 요소는 Hooks입니다. **정책의 자동 집행**입니다."

- "커밋 전에 민감 정보 스캔" — 자동 차단
- "배포 전에 테스트 통과 여부 확인" — 자동 차단
- "고객 데이터 접근 시 로그 기록" — 자동 실행

"이것이 왜 중요할까요? CLAUDE.md는 '지켜주세요'고 Hooks는 '못 하게 막는다' 입니다. 조직에 **강제력 있는 규칙**이 필요할 때 Hooks를 씁니다."
